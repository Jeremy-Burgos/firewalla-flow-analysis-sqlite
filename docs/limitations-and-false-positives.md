# Limitations and False Positives

This workflow is useful, but it should not be oversold.

## High volume is not the same as high severity

A source with a high flow count may simply be noisy.

Volume matters, but it does not answer intent by itself.

## Policy blocks and inbound probing are different signals

This is the biggest practical limitation.

A repeated external source hitting a closed local service is one kind of signal.

A block generated by a DNS or list-based policy is another kind of signal.

They are both useful, but they should not be merged into the same conclusion.

## Direction inference depends on local scope

If your local CIDRs or public application IPs are incomplete, some rows will be misclassified or land in `unknown`.

That is a workflow assumption, not a parser guarantee.

## Country-name rules are not direct geolocation proof

If `Matched By` contains a country name, that tells you what rule fired.

It does not prove precise geolocation by itself.

## Source names can be noisy

Names can change, rotate, or reflect shared infrastructure. For serious review, the IP field usually matters more.

## Export shape matters

The parser is only as reliable as the shape of the export it is built for.

If the CSV columns change, the workflow should be updated instead of forced.

## Public sharing requires discipline

If you later publish findings, sanitize first and review first.

Do not publish raw private telemetry, and do not publish every blocked source as if it were a universally valid threat indicator.
